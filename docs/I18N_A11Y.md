# i18n and Accessibility (AIra)

## i18n

- The app uses **react-i18next** with keys in **src/i18n.ts** (`resources.en.translation`).
- Use `useTranslation()` and `t('key')` for all user-facing strings so the app can be fully localized.
- **Checklist:** Search for hardcoded English strings in JSX (e.g. `"Submit"`, `"Loading..."`) and replace with `t('key')`; add missing keys to `i18n.ts` and set a default/fallback language (e.g. `i18n.changeLanguage('en')` on init).

## Accessibility (a11y)

- **ARIA:** Use `aria-label`, `aria-describedby`, and `role` where they add meaning (buttons that are icon-only, form fields, live regions for toasts/errors).
- **Focus:** Ensure interactive elements are focusable and that focus order is logical; avoid `tabIndex={-1}` unless you manage focus explicitly (e.g. modals).
- **Landmarks:** Prefer semantic HTML (`<main>`, `<nav>`, `<header>`, `<footer>`) and add `role="main"` or `aria-label` for regions when needed.
- **Error boundaries / route errors:** Already wrapped; ensure error messages are announced (e.g. `aria-live="polite"` on the error container).

Quick pass: run an a11y audit (e.g. Lighthouse or axe DevTools) on key flows (login, teaching, settings) and fix critical issues; ensure every UI string goes through `t()`.
